任務分流協作設計

什麼任務交給 Make？什麼任務讓 n8n 處理？設計雙平台協作架構的決策框架，以及三種常見的混合自動化情境。

📖補充參考
⏱閱讀約 8 分鐘
任務分配框架

Make 和 n8n 各有擅長的場景。設計協作架構時，根據任務特性選擇正確的平台，可以最大化兩者的優勢。

MAKE.COM
交給 Make 的任務
  • 接收 Gmail、Line、Slack 等雲端事件
  • 向第三方 API 查資料（天氣、匯率、股價）
  • 更新 Google Sheet、Airtable 等雲端資料庫
  • 發送 Email、Line 通知、Slack 訊息
  • 接收 Webhook 並做簡單過濾轉發
  • 需要 1000+ 種 SaaS 服務連接時
N8N DESKTOP
交給 n8n 的任務
  • 讀寫本機檔案（PDF、Excel、文字檔）
  • 大量迴圈處理（100 個檔案批次改名）
  • 呼叫 Gemini API 做 AI 分析（省費用）
  • 複雜的資料轉換與計算邏輯
  • 敏感資料不想上雲端的任務
  • 需要無限次數執行的排程任務
決策問題清單

每次設計新自動化流程時，問自己這幾個問題：

問題是 → 用 Make否 → 考慮 n8n
需要連接 Gmail / Line / Slack 等 SaaS？Make 有現成模組n8n 可用 HTTP Request 自行串
需要讀寫本機檔案？—n8n 有 File System 節點
需要迴圈處理大量資料（100+）？Make 有次數限制n8n 無限次數
資料敏感，不能上雲端？—n8n 全程在本機處理
一次性設定，不需要維護？Make 介面更直覺—
需要複雜 JavaScript 邏輯？—n8n Code 節點更強大
三種協作情境
SCENARIO 1
Gmail 收發票 → n8n AI 辨識 → 歸檔並回報
Make 監控 Gmail，收到含「發票」附件的郵件觸發
Make 下載 PDF 附件，POST 到 n8n Webhook（含 base64 PDF）
n8n 呼叫 Gemini Vision 辨識發票內容，生成規範檔名
n8n 將 PDF 存入本機並重命名，回傳辨識結果給 Make
Make 發送完成通知到 Line，附上辨識到的廠商與金額
SCENARIO 2
Google Sheet 新增資料 → n8n 批次處理 → 更新回 Sheet
Make 監控 Google Sheet，有新列時觸發（Google Sheets Watch rows）
Make 取得新列資料，POST 到 n8n（含所有欄位）
n8n 執行複雜計算（如從本機資料庫查詢補充資料），生成結果
n8n 回傳結果給 Make
Make 將結果寫回 Google Sheet 對應列（Google Sheets Update row）
SCENARIO 3
定時任務 → n8n 生成日報 → Make 多管道發送
n8n Schedule Trigger 每天 17:30 觸發（不用 Make）
n8n 讀取本機工作記錄，呼叫 Gemini 生成日報
n8n 完成後，HTTP Request 主動 POST 到 Make Webhook
Make 接收日報內容，同時發送到 Gmail、Line 群組、Slack 頻道
💡 設計原則：Make 做「聯絡官」，n8n 做「執行官」。Make 負責對外（雲端 SaaS 的串接與通知），n8n 負責對內（本機的高強度運算與檔案處理）。兩者之間只用 Webhook + HTTP 溝通。
⚠️ 避免過度設計：不是每個流程都需要 Make + n8n 雙平台。如果任務不需要本機檔案系統、不需要大量迴圈、不需要保護隱私，直接在 Make 完成即可。雙平台架構增加了複雜度，只在真正需要時使用。
