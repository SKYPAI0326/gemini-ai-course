Optional Chaining：安全存取巢狀資料

用 ?. 語法讓你的流程在遇到 null 或不存在的欄位時優雅地回傳 undefined，而不是直接崩潰報錯。

⏱預估時間：15 分鐘
📋3 個步驟
1
為什麼需要 ?. ——從一個報錯場景開始

假設你在處理 Google Forms 回傳的資料，有個欄位「公司地址」是選填的。有些人填了，有些人沒填。沒填的時候，JSON 結構中這個欄位可能根本不存在，或者其值是 null。

// 問題情境：地址是選填欄位 // 有人填： { "姓名": "王小明", "地址": { "縣市": "台北市", "區": "中正區" } } // 沒填的人： { "姓名": "陳小華" } ← 根本沒有 "地址" 這個 key // 用普通語法讀地址的縣市： {{ $json.地址.縣市 }} ← ❌ 陳小華的資料會報錯！ TypeError: Cannot read properties of undefined // 用 ?. 安全讀取： {{ $json.地址?.縣市 }} ← ✅ 沒有地址時回傳 undefined，流程繼續執行
2
?. 的使用模式

?. 可以用在物件存取、陣列存取、方法呼叫等各種場景。核心概念是：若問號前的值是 null 或 undefined，整個表達式立即回傳 undefined，不繼續執行後面的存取。

// 物件欄位存取（最常用） {{ $json.聯絡人?.電話 }} {{ $json.地址?.縣市?.郵遞區號 }} ← 可以連用多個 ?. // 陣列元素存取 {{ $json.商品列表?.[0]?.名稱 }} ← 陣列用 ?.[索引] // 在 Code 節點中 const phone = $input.item.json.聯絡人?.電話; const city = $input.item.json.地址?.縣市; const firstItem = $input.item.json.商品列表?.[0]?.名稱; // AI 回傳的巢狀結構（Gemini 回應格式） {{ $json.candidates?.[0]?.content?.parts?.[0]?.text }}
💡 AI 回應格式一定要用 ?.：Gemini API 的回傳結構層次很深（candidates[0].content.parts[0].text），如果某層為 null（例如 AI 安全過濾擋掉了請求），不用 ?. 會整個 Workflow 報錯中斷。
3
?. 與 ?? 搭配使用

?. 讓你安全存取可能不存在的欄位；??（Nullish Coalescing）讓你在值為 null 或 undefined 時提供預設值。兩者搭配，就能同時解決「欄位不存在」和「找不到時給預設值」兩個問題。

// 模式：安全存取 + 預設值 {{ $json.地址?.縣市 ?? '地址未填寫' }} // 實際應用：產出通知郵件時組合收件資訊 const name = $input.item.json.姓名 ?? '使用者'; const phone = $input.item.json.聯絡人?.電話 ?? '未提供電話'; const city = $input.item.json.地址?.縣市 ?? '地址未填'; const message = `${name} 的聯絡電話：${phone}，所在縣市：${city}`; return [{ json: { message } }]; // 讀取 AI 輸出，有安全保護的完整寫法 const raw = $input.item.json.candidates?.[0]?.content?.parts?.[0]?.text ?? ''; if (!raw) { return [{ json: { error: 'AI 未回傳內容' } }]; }
✅ 養成習慣：只要讀取 AI 回傳、表單選填欄位、或任何來自外部 API 的巢狀資料，都用 ?.。這個 10 秒的習慣，能避免 90% 的「流程某天突然崩潰但找不到原因」。
完成驗證 CHECKPOINT
測試 ?. 對 null 欄位的保護

建立測試 Workflow：Set 節點輸出 {"name": "測試", "address": null}，在下一個 Set 節點用 Expression {{ $json.address?.city ?? '未填' }} 讀取城市。執行後應顯示「未填」而非報錯。再把 address 改成有值的物件，應正確顯示城市名稱。
